# 从局部正确到全局错误：AI Code Agent 的可靠性边界与本地开发闭环

## 结论先行

AI Code Agent 擅长根据局部上下文生成看起来合理的修改，但不天然拥有对完整运行时状态机的可靠理解。如果缺少独立验收约束，它还可能使用同一套错误认知验证自己的实现。

因此，提升 AI 编码可靠性不能只依赖一段更长、更严格的 Prompt。Prompt 是提醒，不是安全网。对本地开发更有效的做法是拆分职责：主 Agent 负责实现，挑战 Reviewer 负责寻找反例，本地验收 Agent 负责按固定步骤执行验收。

本文通过一次真实的 iOS 分页容器回归说明问题如何发生，并给出一套可以直接用于本地开发的三角色闭环。

## 案例摘要

个人页原本存在一个问题：用户横滑到其他一级 tab 后，UI 已完成切换，但业务 `selectedIndex` 仍停留在旧 tab，导致下拉刷新命中错误页面。

Agent 在 Parchment 的滚动完成回调中同步 `selectedIndex`，原问题随即消失。但重新启动 App 后，页面默认选中了“灵感”，没有跟随服务端指定的“创作”。

根因是 Agent 同时做了两个错误假设：把回调中的 `pagingItem.index` 当成目标页，并认为该回调只由用户横滑触发。实际上，初始化和程序化选择也会进入该生命周期。一次局部正确的修复因此覆盖了服务端默认选择，造成全局回归。

## 完整因果链

页面初始化原本遵循以下流程：

```text
读取服务端 selected
→ 计算 targetIndex=1
→ 创建页面和菜单
→ reloadData
→ select(targetIndex)
→ 展示创作
```

加入新的滚动回调后，流程变成：

```text
读取服务端 selected
→ 计算 targetIndex=1
→ reloadData / 程序化 select 触发 Parchment 回调
→ 回调错误更新共享 selectedIndex
→ 初始化逻辑继续读取已被覆盖的 selectedIndex
→ 最终展示 index 0 的灵感
```

问题包含两个错误假设：

1. Agent 把 `pagingItem.index` 当成切换后的目标 index，但框架中可靠的目标应由 `destinationViewController` 确定。
2. Agent 假设该回调只由用户横滑触发，忽略了初始化、`reloadData()` 和程序化 `select()` 也可能进入相同生命周期。

最终修复没有硬编码“创作”，而是恢复“服务端 selected 决定默认 tab”的产品约束：

```swift
performInitialTabSelection(targetIndex: targetIndex)
```

初始化使用本次计算得到的局部目标值，不再依赖可能被回调修改的共享状态。横滑完成后，则通过实际目标控制器提交状态：

```swift
guard transitionSuccessful,
      let destinationIndex = items.firstIndex(where: {
          $0.vc === destinationViewController
      }) else {
    return
}
selectedIndex = destinationIndex
```

## AI Code Agent 为什么容易产生这类问题

### 1. 根据表面语义补全，而不是执行状态机

Agent 看到 `didScrollToItem(pagingItem:)`，很容易从命名推断 `pagingItem` 就是目标项。

但真实语义取决于具体框架版本和调用时机：

- 参数代表起始项、当前项还是目标项？
- 初始化时是否触发？
- 程序化切换是否触发？
- 手势取消时是否触发？
- 回调发生在状态提交前还是提交后？

如果不查框架定义或项目内成熟用法，Agent 实际上是在使用命名进行概率推断。

### 2. 局部最优化

Agent 当时观察到的直接问题是：

```text
UI 已经横滑到赞同，但 selectedIndex 仍然是创作。
```

局部目标因此变成：

```text
横滑后更新 selectedIndex。
```

这个目标没有覆盖更大的状态机：

```text
服务端默认值
→ 路由参数
→ 初始化
→ reloadData
→ 程序化 select
→ 用户点击
→ 用户横滑
→ 下拉刷新
```

于是出现了典型的“局部正确、全局错误”：新代码解决了横滑同步，却破坏了冷启动默认选择。

### 3. 实现和验证共享同一个盲区

生成代码和设计验证的是同一个 Agent。它认为主要风险是横滑白屏、loading 不结束和刷新命中错误 tab，因此验证也集中在这些路径。

它没有意识到初始化生命周期已经被影响，自然也没有主动增加以下用例：

```text
彻底重启 App
→ 获取服务端 selected
→ 检查默认高亮和实际内容
```

Agent 不知道自己误解了什么，因此也容易遗漏能够揭示该误解的测试。

### 4. 看到正向证据后过早收敛

横滑后的刷新日志从旧 tab 变成新 tab，证明新增回调对原问题有效。Agent 随后容易形成“修复完成”的判断，而没有继续追问：

- 初始化是否也经过这个回调？
- 回调参数是否真的表示 destination？
- 完整重启后服务端 selected 是否仍生效？
- 当前验证使用的 Mock 是否真的处于启用状态？

这是一种由局部成功证据引发的确认偏差。

## 为什么只增加 Prompt 不够

Prompt 可以要求 Agent 更谨慎，但它仍是概率性约束。

即使 Prompt 写着“检查所有边界情况”，Agent 仍需自己判断什么是边界情况。如果它误解了框架语义，就可能在错误认知下完成检查，并认为已经满足要求。

本地开发中需要解决的不是“如何写一个覆盖所有未知问题的 Prompt”，而是“如何避免同一个 Agent 同时实现、出题和判卷”。三角色闭环通过认知隔离降低共同盲区：

| 角色 | 核心职责 | 不负责什么 |
| --- | --- | --- |
| 主 Agent | 定位根因、修改代码、提供初步证据 | 不独自宣布最终通过 |
| 挑战 Reviewer | 基于 diff 主动寻找反例 | 不修改代码，不定义产品预期 |
| 本地验收 Agent | 按固定清单操作设备并报告结果 | 不解释实现，不临时降低标准 |

## 本地开发的三角色闭环

### 角色一：主 Agent 负责实现

主 Agent 开始编码前只需要形成一份简短的变更契约：

```text
目标行为：横滑成功后刷新命中新 tab
预计改动：delegate 回调、selectedIndex 写入
允许变化：横滑后的业务 selectedKey
必须保持：冷启动、点击切换、reload 后当前选择
验收关系：selectedKey == destinationKey；reloadAfter == reloadBefore
```

然后完成四件事：

1. 记录原问题的日志或截图。
2. 使用 LSP 查改动状态的定义、写入和关键读取。
3. 查项目内同一框架回调的成熟用法。
4. 做最小修改并执行本地构建。

主 Agent 可以提供初步验证，但它的验证只说明“实现者认为修改可用”，不能作为最终结论。

### 角色二：挑战 Reviewer 负责寻找反例

挑战 Reviewer 接收变更契约、实际 diff 和主 Agent 的证据。它不沿着主 Agent 的结论继续证明，而是从相邻入口主动构造反例：

```text
初始化是否触发同一回调？
程序化 select 和用户横滑是否共用路径？
横滑取消是否提前写入状态？
reloadData / reloadMenu 是否造成回调重入？
异步返回时用户是否已经切换页面？
```

统一 Prompt：

```text
请基于变更契约、实际 diff 和运行证据寻找反例。
不要修改代码，不要证明当前实现正确，也不要自行发明产品预期。
每个问题必须包含：触发条件、违反的基线或关系、可观察证据、严重程度。
无法确认的产品行为标记为“需要用户决策”。
```

挑战 Reviewer 可以运行针对性操作证明反例成立，也可以确认对应问题修复后不再出现，但它不负责最终验收。

### 角色三：本地验收 Agent 负责判定结果

本地验收 Agent 是独立角色。它只接收验收清单、运行命令和输入数据，不读取主 Agent 的实现解释，避免被“为什么这段代码应该正确”所锚定。

它在本机执行现有工具，例如：

```text
docs/debug.sh all：编译、安装并拉起 App
agent-device：点击、横滑、下拉刷新、截图和读取 UI 状态
Whistle：切换固定接口响应
本地日志：核对 selectedKey、刷新目标和 loading 状态
```

针对本次问题，验收清单是：

| 场景 | 操作 | 判断关系 |
| --- | --- | --- |
| 冷启动 | 读取服务端 selectedKey 后重新进入页面 | UI selectedKey == serverSelectedKey |
| 点击切换 | 点击任意一级 tab | UI、业务 index、实际 VC 一致 |
| 横滑成功 | 横滑到相邻 tab | selectedKey == destinationKey |
| 横滑取消 | 拖动后返回原 tab | selectedKey == beforeSelectedKey |
| 页面刷新 | 在当前 tab 下拉刷新 | afterSelectedKey == beforeSelectedKey |
| Mock 往返 | 切换 tab 接口响应后刷新 | 不白屏、不残留 loading、当前 key 保持 |

验收 Agent 的输出只能是：

```text
通过：所有清单项都有本地证据
失败：给出失败步骤和观察结果
未验证：说明无法执行的项目和原因
```

它不能看到部分成功后把“未验证”解释成“应该没问题”。

## 三个角色如何配合

本地开发的完整顺序如下：

| 步骤 | 负责人 | 输出 |
| --- | --- | --- |
| 1. 复现并建立变更契约 | 主 Agent | 原问题证据、允许变化、必须保持项 |
| 2. 实现最小修复 | 主 Agent | diff、构建结果、初步证据 |
| 3. 构造反例 | 挑战 Reviewer | 阻断问题、残余风险、需要决策项 |
| 4. 修复阻断问题 | 主 Agent | 更新后的 diff 和证据 |
| 5. 独立本地验收 | 验收 Agent | 通过、失败或未验证 |
| 6. 处理未知产品行为 | 用户 | 明确预期后重新验收 |

挑战 Reviewer 和验收 Agent 是两个角色：

- Reviewer 负责扩大搜索范围，回答“还可能在哪里错”。
- 验收 Agent 负责执行固定清单，回答“约定的行为是否真的成立”。

对于只改文案或局部样式的低风险任务，可以不启动两个额外 Agent。对于共享状态、生命周期、并发、缓存和核心交互，建议三个角色都保留。

## 本地完成门槛

主 Agent 只有在以下条件全部满足后才能结束任务：

- 原问题有复现证据，修复后证据消失。
- 实际 diff 与变更契约一致，没有未解释的额外修改。
- 挑战 Reviewer 没有未处理的阻断问题。
- 本地验收 Agent 的关键清单全部通过。
- 无法执行的项目明确标记为“未验证”。
- 未确认的产品行为已经由用户决策。

这个门槛只使用已有本地工具，不要求新增基础设施。三个角色各自完成单一职责即可。

## 最后的原则

不要让同一个 AI Code Agent 同时拥有三种权力：

```text
定义什么叫正确
实现代码
判断自己是否通过
```

更可靠的本地分工是：

```text
主 Agent 实现
挑战 Reviewer 寻找反例
本地验收 Agent 执行清单
用户决定未知的产品行为
```

AI 编码的价值不在于永远不犯错，而在于以更低成本完成实现。本地开发流程的责任，是通过角色隔离和运行证据及时发现错误，而不是依赖实现者自己意识到它误解了什么。
