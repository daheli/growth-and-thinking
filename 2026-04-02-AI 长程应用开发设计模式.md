AI 长程应用开发设计模式

原链接: https://www.anthropic.com/engineering/harness-design-long-running-apps

anthropic 的博文《Harness design for long-running application development》揭示了如何让 AI 智能体（Agent）像经验丰富的软件工程师一样，能够持续数小时甚至数天处理复杂的全栈项目，而不至于因上下文丢失或逻辑疲劳而崩溃。

其核心思想是：与其追求一个具备超长记忆的“天才”模型，不如构建一个能够自我管理、自我记录的“工程制度”环境（Harness）。

------

### 一、 核心重点（The Core Philosophy）

文章强调了长程 AI 任务中的三个关键瓶颈：上下文污染（过多的历史信息让 AI 抓不住重点）、时间盲区（AI 不知道自己忙了多久，容易陷入死循环）和判断力退化。

为解决这些问题，Anthropic 提出了以下设计要点：

1. **分工明确的“三智能体”架构：**
2. - **规划者（Planner）：** 负责将大目标拆解为可落地的 JSON 格式任务清单。
   - **执行者（Generator）：** 每次只盯着一个任务，完成后通过 Git 提交代码并更新进度文件。
   - **评估者（Evaluator）：** 独立角色，负责对结果进行打分和反馈，防止执行者“自卖自夸”。
3. **强制的上下文重置（Context Resets）：** 与其让对话历史无限延长，不如在完成一个小任务后“刷新”对话，让 AI 只带上最新的代码状态和进度报告（如 `claude-progress.txt`）重新开始。
4. **结构化手递手（Structured Handoffs）：** 使用 Git 提交信息和 JSON 格式的任务列表作为 Agent 之间的“记忆”。即使 Agent 换了一轮，新上岗的 Agent 只要读一下 Git log 和任务列表，就能立刻上手。

### 二、 普通程序员如何落地实施？

你不需要复现 Anthropic 内部昂贵的系统，普通程序员在日常开发中借鉴其思路，可以显著提升使用 LLM（如 Claude 3.5/4.5 或 GPT-4o）编写大型项目的成功率：

#### 1. [建立“项目契约”文件（CLAUDE.md](http://建立“项目契约”文件（CLAUDE.md) / PROGRESS.md）

在项目根目录下手动或让 AI 创建一个 [`CLAUDE.md`](http://CLAUDE.md) 文件。

- **做法：** 记录项目的技术栈、代码风格、关键决策以及当前的进度。
- **收益：** 当你开启一个新的对话窗口（New Chat）时，直接把这个文件投喂给 AI，它能瞬间找回状态，避免从头解释项目背景。

#### 2. 贯彻“单任务、单提交”原则

不要对 AI 说：“帮我把这个电商网站的后台和前端都写了”。

- **落地：** 1. 让 AI 先写一个 `todo.json`。 2. 指令示例：“阅读 todo.json，完成任务 #1。完成后运行测试，并通过 git commit 提交，最后更新 `todo.json` 的状态。” 3. 每完成一个子任务，**新开一个对话**，把最新的代码和 `todo.json` 传给它。

#### 3. 引入“测试驱动”的自动化反馈

AI 容易写出“看起来正确但跑不通”的代码。

- **落地：** 在提示词中规定：任何代码修改必须伴随运行特定的测试脚本（如 `npm test`）。
- **技巧：** 如果测试失败，要求 AI 必须将错误日志写入 `error.log`。由于 AI 处理文件比处理对话气泡更精准，这种“物理隔离”的错误记录能帮它更快 debug。

#### 4. 使用 Git 轨迹管理 AI

把 Git 当作 AI 的“长期记忆”。

- **落地：** 强制要求 AI 在每次修改后生成详细的 Commit Message。
- **价值：** 如果 AI 把代码改乱了，你可以根据其 Commit 记录（它的思维路径）迅速回滚，或者通过 diff 告诉它哪一部分理解偏了。

### 总结

Anthropic 的方案告诉我们：要把 AI 当成一个“会失忆但很勤奋”的初级程序员。 程序员的职责不再是敲每一行代码，而是通过构建一套标准的文档流、任务流和测试流，为 AI 提供稳定的工作环境。
