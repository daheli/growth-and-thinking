# Harness Engineering 就是控制论：从蒸汽机到 AI 写代码，同一个模式反复出现

原文: https://mp.weixin.qq.com/s/qdgH3Ha27HZchoP3bl8WJg?scene=1

最近反复读了很多遍 George（@odysseus0z[1]）写的这篇 Harness Engineering Is Cybernetics[2]

这篇文章把 OpenAI 的 harness engineering[3] 和两百多年前的蒸汽机调速器放在一起讲，乍一看跨度很大，但读完之后你会发现，**它们本质上就是同一件事**。

如果你对 **AI 编程、工程管理或者技术趋势** 感兴趣，强烈建议阅读 OpenAI 的 原文[3] 和 George 的 原文[2]。

这里我把其中 **最核心的观点** 整理出来，再补充一些控制论的背景知识，尽量用通俗的方式聊一聊。

## 一个反复出现的模式

George 在读 OpenAI 关于 harness engineering 的文章时，有一种 **似曾相识的感觉**。仔细想了想，他发现这个模式他至少见过三次。

第一次是 1780 年代，詹姆斯·瓦特发明了离心调速器。

在那之前，蒸汽机旁边必须站一个工人，时刻盯着转速，手动调节阀门。转快了就关小一点，转慢了就开大一点。这活儿 **枯燥、重复，但又不能不做**。

瓦特设计了一个飞球机构，它能自动感知转速，转速高了飞球就甩开、阀门关小，转速低了飞球就收回、阀门开大。

工人不用再手动拧阀门了，但工人并没有消失，他的工作变成了 **设计和调试这个调速器本身**。

下面这张图展示的就是瓦特的离心调速器：

![Image](https://mmbiz.qpic.cn/mmbiz_png/JGVtJ5g0NRLZLawatzaCK9gL4ZOAyMkX5BDc0czt0S2ICdzYiaHP931SdXvOKmM3MJLK2Wl6I50srcUUmgvCYpTjzZhZzwtiaZsjT2UXPAvPQ/640?wx_fmt=png&tp=webp&wxfrom=5&wx_lazy=1#imgIndex=0)

第二次是 Kubernetes。做过运维或者后端开发的同学可能都接触过，Kubernetes 的核心思想是 **声明式管理**。

你不需要告诉系统每一步该怎么做，你只需要描述你想要的最终状态：我需要三个副本，用这个镜像，给这么多资源。

然后 Kubernetes 的控制器会持续观察实际状态，发现跟你期望的不一样，就自动修复。容器挂了就重启，副本少了就补，部署出问题了就回滚。

工程师的工作从手动重启服务变成了 **编写那份描述期望状态的规范文件**。

第三次就是现在。OpenAI 描述了一种新的工程模式：工程师不再直接写代码，而是设计环境、构建反馈回路、把架构约束写成规则，然后让 AI agent 去写代码。

五个月内产出一百万行代码，**没有一行是人手写的**。他们管这叫 harness engineering。

这三次变化，跨越了两百多年，但 **核心模式是一样的**。

你从亲手干活的人，变成了 **设计控制系统的人**。

## 先聊聊控制论到底是什么

在继续往下说之前，有必要先铺垫一下控制论这个概念。很多人可能没有专门学过控制论，但其实它的核心思想并不复杂。

1948 年，数学家诺伯特·维纳出版了一本书，正式提出了控制论（cybernetics）这个学科。这个词来自希腊语 κυβερνήτης，意思是舵手。

有意思的是，Kubernetes 这个名字也来自同一个希腊词根。维纳想表达的就是：**无论你控制的是一艘船、一台机器还是一个组织，背后的原理都是相通的**。

控制论的核心思想可以用一句话概括：**通过反馈来实现目标**。

听起来很简单，但它包含几个关键的组成部分，理解了这些，后面的内容就会清晰很多。

第一个是目标（Reference / Desired State）。任何控制系统都需要有一个明确的目标，就是你希望系统达到的状态。家里的空调设定 26 度，这个 26 度就是目标。

第二个是传感器（Sensor）。系统需要有办法感知当前的实际状态。空调里的温度计就是传感器，它告诉系统现在房间多少度。

第三个是比较器（Comparator）。把实际状态和目标做对比，找出差距。现在 30 度，目标 26 度，差了 4 度，这个判断过程就是比较。

第四个是执行器（Actuator）。根据差距采取行动，让实际状态向目标靠拢。空调的压缩机就是执行器，它负责制冷把温度降下来。

第五个，也是最关键的，是 **反馈回路（Feedback Loop）**。执行器采取行动之后，传感器会再次感知新的状态，然后再比较、再行动。

这个过程是 **持续循环的，不是做一次就停了**。正是这个不断循环的机制，让系统能够自动趋近目标并保持稳定。

用最直观的方式来理解：你开车想保持 100 公里的时速，眼睛看速度表（传感器），发现速度掉到 95 了（比较），脚踩一下油门（执行器），速度又回到 100，然后你继续盯着速度表（反馈回路）。

如果你开的是有定速巡航的车，这个过程就 **完全自动化** 了，你只需要设定目标速度就行。

无论是舵手掌舵、调速器控制蒸汽机、Kubernetes 管理容器，还是 harness engineering 让 AI 写代码，做的都是同一件事：**你不再亲自去转那个阀门，你去掌舵**。

每一次这个模式出现，都是因为有人造出了 **足够强大的传感器和执行器**，能够在那个层面上闭合反馈回路。

瓦特造出了能感知转速并自动调节阀门的机械装置。Kubernetes 造出了能感知集群状态并自动调和的软件控制器。而现在，LLM 成了能在代码层面感知和执行的新工具。

## Harness Engineering 里的控制论骨架

理解了控制论的基本组成部分之后，我们来看看 harness engineering 里这些要素分别对应什么。

目标（Reference）对应的是你为代码库定义的 **质量标准和架构规范**。架构文档里描述的分层方式、依赖方向、设计原则，还有那些编码了团队审美的 golden principles，这些加在一起构成了系统的目标状态。

就像空调设定的 26 度一样，它告诉整个系统 **什么样的代码才算达标**。

传感器（Sensor）和比较器（Comparator）在控制论教科书里是两个独立的部件：传感器负责感知当前状态，比较器负责拿着标准判定达没达标。

但在 harness engineering 的实际工具中，这两个角色 **几乎总是合在一起的**。测试套件执行代码拿到实际输出，这是感知；紧接着 assert 语句就把实际输出和期望值做对比，这是比较。

Linter 扫描代码是感知，同时内置的规则判定违规就是比较。CI 管道跑完所有检查后给出通过或失败的最终裁决，一条龙地完成了从感知到比较的全部工作。

概念上是两件事，但实现上往往封装在同一个工具里，就像家里的恒温器把温度计和比较电路装在了同一个盒子里一样。

理解这一点很重要：你在构建 harness 的时候，既要考虑怎么让系统 **感知到代码的当前状态**，也要考虑用什么标准去 **判定好坏**，即使最终这两件事是同一个工具在做。

执行器（Actuator）对应的就是 AI agent 本身。它根据检测到的偏差信号采取行动：修复失败的测试、重构不合规范的模块、调整不一致的接口。

值得注意的是，LLM 在 harness engineering 中其实 **身兼多职**。它既能作为传感器去感知代码质量，比如阅读代码后判断架构风格是否一致；又能作为比较器，对照架构文档判定代码是否符合设计意图；还能作为执行器去动手修改代码。

正是因为同一个 LLM 能同时覆盖 **感知、比较和执行** 这三端，才打通了以前打不通的那个回路。

反馈回路（Feedback Loop）就是把以上所有环节串起来的那个持续循环。agent 写代码，各种检测机制感知并判定偏差，agent 再修改，如此反复，直到代码达到目标状态。

这个循环 **不需要人来驱动**，它是自动运转的。

这样一拆解你就会发现，harness engineering **完全就是一个标准的控制论系统**。

工程师的角色也变得清晰了：你不是那个亲手拧阀门的人，你是那个 **设定目标、校准检测机制、调试整个反馈回路的人**。

## 其实没那么复杂

不过话说回来，上面用控制论的五个组件去逐一对应 harness engineering，可能有点把简单的事情复杂化了。

实际上 harness engineering 的核心可以用 **一张图讲清楚**。

Anthropic 在他们那篇经典的 Building Effective Agents[4] 中给 Agent 画了一张定义图：

![Image](https://mmbiz.qpic.cn/mmbiz_jpg/JGVtJ5g0NRLHdibhedMFAQedYZiafdibjOWvnSDBziap4Nstcs84bPD95mKyhB8FT1ARzkicicGoPF0ibCnicu1T6CHhFymyuSlWTdA3lWdbdRz8Sks/640?wx_fmt=jpeg&tp=webp&wxfrom=5&wx_lazy=1#imgIndex=3)

这张图的结构非常简单：LLM 向环境（Environment）发出动作（Action），环境把结果以反馈（Feedback）的形式返回给 LLM，然后 LLM 根据反馈决定下一步动作，如此循环，直到任务完成（Stop）。

人类（Human）在旁边，负责 **启动任务和必要时介入**。

Harness engineering 的本质就是这个循环。Agent 写代码，环境给反馈，agent 再改，再拿反馈，直到通过。

所谓 harness，就是你 **精心设计的那个环境**：你往里面放什么样的测试、什么样的检查规则、什么样的架构文档，决定了反馈的质量，也就决定了 agent 最终产出的质量。

控制论的那些概念，目标、传感器、比较器、执行器、反馈回路，本质上都藏在这个简单的 **Action → Feedback 循环** 里。

只是当你需要更精细地思考怎么设计这个环境的时候，把它们拆开来看会更有帮助。

## 代码为什么是最后一个被攻克的领域

这就引出了一个很关键的问题：既然这个模式这么好用，为什么代码领域直到现在才出现？

其实代码领域一直有反馈回路，只是停留在比较低的层面。编译器能检查语法对不对，测试套件能检查功能对不对，代码检查工具能检查风格规范对不对。

这些都是真正的控制论式控制，但它们只能处理 **机械性的检查**：能不能编译？测试通不通过？格式符不符合规范？

但更高层面的判断呢？比如这个改动符不符合整个系统的架构？这个方案是不是最合适的？这个抽象方式会不会随着代码库增长带来问题？

这些事情过去没有任何自动化工具能搞定，只有人能判断，也只有人能动手修改。**感知和执行两端都被人类独占着**。

这就好比一个工厂里，流水线上的质检可以用机器来做，但产品设计好不好、市场方向对不对，这些判断只能靠人。

代码领域也是类似的情况，底层的检查早就自动化了，但高层的 **架构判断和设计决策** 一直是人力活。

## LLM 同时改变了两端

LLM 的出现打破了这个僵局，而且是一次性打破的。它同时改变了 **感知端和执行端**。

在感知端，LLM 能理解代码在架构层面是不是合理的，某个模块的设计是否一致，某个接口是不是跟系统其他部分的风格冲突。

这些过去只有经验丰富的工程师才能看出来的问题，LLM 现在也能识别。

在执行端，LLM 能动手改。它能重构一个模块，重新设计一个不一致的接口，围绕真正重要的契约重写测试套件。

这些同样过去只有人能做的事情，LLM 现在也能做了。

这意味着反馈回路终于可以在 **真正重要的决策层面闭合** 了。不光是语法对不对、测试过不过，而是架构合不合理、设计好不好。**这是一个质的跳跃**。

但闭合回路只是必要条件，还远远不够。瓦特的调速器需要调校，Kubernetes 的控制器需要正确的规范文件，而让 LLM 在你的代码库上工作，需要提供的东西更难。

## 让你的判断力变成机器能读懂的东西

基本的反馈回路搭起来并不难。让 agent 能跑测试、CI 输出能被解析、错误信息能指向修复方向，这些是基础功课。

Nicholas Carlini 做过一个实验，用 16 个并行的 agent 构建了一个 C 编译器，用的 prompt 非常简单，但他精心设计了整个测试基础设施。

他自己说过，**大部分精力花在了设计 agent 周围的环境上**，包括测试、环境和反馈机制。

更难的部分在于，你要用你系统特有的知识来 **校准这个传感器和执行器**。这也是大多数人卡住的地方。

很多人用 AI 编程工具的时候会抱怨：它老是做错，它不理解我们的代码库。George 指出，这个诊断几乎总是错的。

agent 失败通常不是因为它能力不够，而是因为它需要的知识被 **锁在你脑子里**，你从来没有把它写出来。

什么叫好代码？你的架构鼓励什么样的模式？避免什么样的模式？这些如果没有文档化，agent 第一次会犯的错，第一百次还会犯。agent 不会通过耳濡目染来学习。

这个观察非常深刻。想想我们在团队里带新人的经验。一个新人加入团队，如果团队的规范、设计原则、代码风格都只存在于老员工的脑子里，口口相传，那新人就得花很长时间才能摸到门道，中间会犯很多本可以避免的错误。

AI agent 也是一样的。区别在于，新人好歹还能通过观察和提问慢慢学，但 agent **每次运行都是从零开始的**。

所以真正的工作是把你的判断力变成 **机器可以读取的形式**。

包括描述实际分层和依赖方向的架构文档，内置了修复指引的自定义代码检查规则，编码了团队审美标准的核心原则。

OpenAI 自己的经验也印证了这一点：他们发现每周五有 20% 的时间花在清理 AI 产生的低质量代码上，直到他们把自己的标准编码进了 harness 本身，问题才得到解决。

## 那些正确的做法，代价终于变得不可承受了

文章中有一个观点特别值得深思：harness engineering 所要求的那些实践，文档、自动化测试、把架构决策写成规则、快速反馈回路，这些东西其实每一本工程书籍都在推荐。

过去三十年的软件工程经典里，你都能找到这些建议。

但大多数人跳过了这些步骤。为什么？因为过去不做这些事情的代价是 **缓慢而分散的**。

代码质量慢慢下滑，新人上手越来越痛苦，技术债在暗处悄悄积累。痛归痛，但还能忍，还能凑合。

AI agent 的出现让这个代价变得极端了。你不写文档，agent 就会无视你的规范。不光是一个 PR 的问题，而是 **每一个 PR 都会出问题**，以机器的速度，全天候地出问题。

你不写测试，反馈回路根本就闭合不了，agent 写出来的东西你都没法验证对不对。

你不把架构约束写清楚，架构漂移的速度会比你修复的速度快得多。

更要命的是，如果你没有做好这些基础工作，你甚至没法用 agent 来收拾这个烂摊子。因为 agent 自己都不知道 **什么样子才叫干净**。

制造问题的机器同样没法解决问题，除非你告诉它 **什么才是对的**。

这段话让我想到一个很朴素的道理。很多我们一直知道应该做、但因为短期成本而偷懒跳过的事情，迟早会以更大的代价回来找你。

过去这个代价是温水煮青蛙式的，你感觉不到疼。但当 AI 把一切都加速了之后，那些偷过的懒会以 **十倍百倍的速度放大**。

反过来说，那些扎扎实实做好基础工作的团队，在 AI 时代反而会获得 **巨大的放大收益**。

## 你不需要比机器写得更快，你需要比它看得更准

文章最后提到了一个计算机科学中著名的直觉：**生成一个正确的解比验证一个解要难得多**。这就是 P 与 NP 问题背后的思想，也被 LLM 相关的研究实证地验证过。

![Image](https://mmbiz.qpic.cn/mmbiz_jpg/JGVtJ5g0NRJ7k1tIocjGc0bPUq9icByTh6I5hTXcNpxib7QAiaUseaW9AxR47TU5D96oRrnIh2e5PvrP4b0u7uAukvLeZ1Z6FqkqicSKwk9229w/640?wx_fmt=jpeg&tp=webp&wxfrom=5&wx_lazy=1#imgIndex=4)

把这个思想放到工程实践中，意味着什么？你不需要在写代码这件事上胜过 AI，因为它可以全天候地、并行地写代码，这场比赛你赢不了。

你需要做的是 **比它更会评判**：能够清晰地定义什么叫做正确，能够识别出输出哪里有问题，能够判断方向对不对。

这其实跟很多行业的演进是类似的。一个优秀的电影导演不需要自己会演、会拍、会剪、会做特效，但他需要有 **极好的判断力**，知道什么是好的，知道什么需要改。

一个优秀的产品经理不需要自己写代码，但他需要准确地定义需求，需要能判断产品的方向是不是对的。

未来的工程师也会越来越像这样。你的核心竞争力不再是手速有多快、语法有多熟，而是你的判断力有多精准：你能不能定义清楚什么是好的？你能不能建立起一套系统，让 AI 按照你的标准持续产出高质量的结果？

## 回不去了

George 在文章最后说了一句意味深长的话：当年设计瓦特调速器的工人，后来并没有回去继续手动拧阀门。不是因为他们拧不了，是因为那样做已经没有意义了。

这句话放到今天的语境里同样适用。一旦你学会了设计反馈回路、编码架构约束、让 AI 在你搭建的轨道上运行，你就不会想回到逐行手写代码的状态了。

就像你学会了用导航开车，你不会想回到拿着纸质地图一个路口一个路口对照的日子。

Harness engineering 听起来是个新概念，但内核是一个 **有两百多年历史的老思想**：你从执行者变成掌舵者，从亲自动手变成设计控制系统。

这个转变在蒸汽机时代发生过，在云计算时代发生过，现在又在 AI 编程时代发生了。

对于每一个还在写代码的人来说，最值得思考的可能是这件事：与其纠结 AI 会不会取代程序员，不如想想怎样让自己从 **拧阀门的人变成设计调速器的人**。

把你脑子里那些只可意会的经验、判断和标准，变成写下来的、可执行的、机器能理解的规则。这才是在 AI 时代真正有价值的能力。
